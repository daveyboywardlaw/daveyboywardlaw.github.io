Now you may or may of heard of a type os testing called 'happy path' testing.

In Michael Boltons blog "Blog: Very short blog posts (32) The Happy Path"  he states that:


“Happy path testing” isn’t really testing at all. Following the “happy path” is a demonstration.

Now I agree that happy path is a demonstration however I disagree with the statement that it isn't really testing. Providing that the happy path provides me with information about the system then to me it is testing. For this blog post I am going to assume that happy path testing occurs and it involves a tester going though a very simple/standard scenario to make sure that there is no major issues with the software they are testing. Often these may be the first set of tests that you run after you have received a new release, and this is what I usually do - you can argue the merits of this approach but I have found quite a few bugs this way. Hey, for me its a heuristic :)

What do I mean by Users Happy Path?
We all test differently and when we do test it may be that we don't use the system the same way as the users. That is why we test using personas and it is also why testing is difficult. One reason foe this is because all users have there own views and options on what makes a quality software product and they often have their own little nuances that they perform when interacting with a system.  When performing a task on a system the users happy path will probably be different to a testers happy path.  

Imagine the scenario....

You are testing a simple form that accepts your personal details.

Now you may perform this test by:
  • Enter your first name
  • Click into the Last Name field
  • Enter your last name
  • Click on the address line 1 field
  • Enter line 1 of your actual address
  • Click on the Town field
  • Enter the town where you live
  • Click on the Postcode/Zip field
  • Enter your postcode or ZIP
  • Click the Submit button

Now to you that may make a lot of sense. It's a simple test where you are entering data and once entered, that data is then submitted off somewhere. Now imagine you are an actual user of the system. They may test it as follows:

  • Click on the address line 1 field
  • Enter line 1 of your actual address
  • Tab to the Town field
  • Enter the town where you live
  • Tab back to the last name field
  • Enter their last name
  • Tab back to the first name field
  • Enter their first name
  • Tab to the Postcode/ZIP field
  • Enter their postcode/ZIP
  • Tab to the submit button and press enter
Now If everything is working OK you would expect the same result for both tests. However, you are not entering data like the user does so wouldn't it be better to enter the data like they do?

Well of course it would.... :) 

As testers we are not users and we are not quality gatekeepers. We are people who provide information about a product which in turn allows people to make an informed decision about whether the software should be shipped to the customer or not. Now by running these tests the same way as a user will potentially give you the following:
  • An idea that the system will work in the 'correct' way when the actual users start using it
  • The ability to spot unexpected bugs
So how do you find out how a user interacts with the system?
Well this can be the difficult bit. Maybe you are in a position to spend time with the users and see how they use the system or maybe you have some analytics you can use. This would require some resource and some time but the benefits to your testing can be worth the effort.

So when you do happy path/smoke tests try and test like a user. You really should always try and test like a user but that is not always possible. For the really interesting, edge case bugs you find you will probably not necessarily be using the system like the majority of users because if you did you would of probably found it during the happy path.
